# LocalPdfReader V0.1 源代码阅读与逐模块调试指南

## 1. 使用目标

这份指南不是要求一次读完所有代码，而是把 V0.1 拆成可以单独理解和调试的小模块。每一轮只解决三个问题：

1. 输入从哪里来；
2. 数据经过哪些对象和方法；
3. 输出、错误和资源由谁处理。

建议每轮控制在 30～60 分钟。先运行测试观察纯业务逻辑，再启动界面观察完整调用链。不要一开始同时调试界面、命名管道、PDFium 和翻译网络请求。

## 3. 先理解项目依赖方向

```text
App ──────────────→ Application ─────→ Domain
 │                       │              ↑
 └──────────────→ Infrastructure ───────┤
                         │              │
                         └→ PdfProtocol ┘

PdfWorker ─────────────→ PdfProtocol ──→ Domain
```

核心规则：

- `Domain` 只描述数据，不知道 WPF、HTTP、Windows 或 PDFium。
- `Application` 定义业务接口和规则，不直接操作 PDFium、Windows 凭据或 HTTP。
- `Infrastructure` 实现这些接口。
- `App` 负责界面、命令和组合对象。
- `PdfWorker` 是另一个进程，只有它调用 PDFium。
- `PdfProtocol` 是两个进程都认识的消息语言。

如果调试时发现 `App` 直接调用 PDFium，或 `PdfWorker` 读取 API 密钥，就说明破坏了既定边界。

## 4. 调试前准备

### Visual Studio 建议设置

1. 用 Visual Studio 打开 `LocalPdfReader.sln`。
2. 将 `LocalPdfReader.App` 设为启动项目。
3. 使用 `Debug` 和 `x64`。
4. 在“工具 → 选项 → 调试”中关闭“仅我的代码”会看到更多框架内部调用；初学阶段建议保持开启，避免进入 .NET 内部。
5. 打开“异常设置”，先勾选“公共语言运行时异常”中的“用户未处理”，不要一开始勾选所有“引发时中断”。
6. 准备一个普通、有文字层、页数较少的 PDF，路径中最好不包含隐私信息。

### 常用调试窗口

- 局部变量：查看当前方法的输入与临时值。
- 自动窗口：查看当前行附近使用的对象。
- 监视：长期观察 `DocumentId`、`PageIndex`、`ZoomFactor`、`Rotation`、`RequestId`。
- 调用堆栈：回答“是谁调用了当前方法”。
- 线程和任务：分析异步操作，但第一轮不需要深入。
- 输出窗口：查看构建、调试和日志信息。

### 断点操作

- `F9`：添加或删除断点。
- `F5`：继续运行。
- `F10`：逐过程，不进入被调用方法。
- `F11`：逐语句，进入被调用方法。
- `Shift+F11`：跳出当前方法。
- 条件断点：右键断点，例如设置 `request.PageIndex == 2`，避免每一页都暂停。

## 6. 第一轮：领域模型

### 阅读文件

1. `Domain/DocumentId.cs`
2. `Domain/PdfDocumentModels.cs`
3. `Domain/PdfTextModels.cs`
4. `Domain/TranslationModels.cs`

### 重点理解

- `record` 适合表达不可变的数据快照。
- `DocumentId` 防止把不同文档的请求混在一起。
- `PageRenderRequest` 包含一次渲染所需的页码、缩放、旋转和质量。
- `RenderedPageDescriptor` 描述共享内存中的位图，而不是直接携带整张图片。
- `TextGlyph` 是一个字符及其 PDF 页面坐标。

### 调试练习

在 `tests/LocalPdfReader.Domain.Tests/ProjectSmokeTests.cs` 的测试方法中下断点，使用“调试测试”。观察构造出的 `DocumentId` 和记录类型，不启动 WPF。

### 完成标准

能够用自己的话说明 `DocumentId`、`PageIndex`、`RequestId` 分别解决什么问题。

## 8. 第三轮：命名管道协议

### 阅读顺序

1. `PdfProtocol/HandshakeProtocol.cs`
2. `PdfProtocol/PipeMessageEnvelope.cs`
3. `PdfProtocol/PipeMessageSerializer.cs`
4. `tests/LocalPdfReader.PdfProtocol.Tests/ProjectSmokeTests.cs`

### 重点理解

- Envelope（消息信封）保存消息类型、RequestId、DocumentId 和 JSON 负载。
- RequestId 让并发请求能够找到自己的响应。
- 长度前缀告诉接收端一条 JSON 消息有多少字节，避免把半条或多条消息误当成一条。
- Handshake（握手）先确认主程序和 Worker 的协议版本一致。

### 推荐断点

- `PipeMessageSerializer.WriteAsync()`：观察长度前缀和序列化后的字节数。
- `PipeMessageSerializer.ReadAsync()`：观察如何完整读取一条消息。
- `SerializePayload()` 和 `DeserializePayload<T>()`：观察具体请求如何进入信封。

### 完成标准

能够画出 `RenderPageRequest → PipeMessageEnvelope → RenderPageResponse` 的往返过程。

## 9. 第四轮：PDF Worker 与 PDFium

### 阅读顺序

1. `PdfWorker/Program.cs`
2. `PdfWorker/WorkerDocumentService.cs`
3. `PdfWorker/PdfNativeAdapter.cs`

### 调试 Worker 的正确方式

Worker 是由主程序动态启动的第二个进程。普通 F5 只会直接调试主程序，Worker 断点可能不会自动命中。

1. 在 `Program.cs` 的请求 `switch` 中下断点。
2. F5 启动主程序并等待界面出现。
3. 在 Visual Studio 选择“调试 → 附加到进程”。
4. 选择 `LocalPdfReader.PdfWorker.exe`，代码类型选择托管 .NET 代码。
5. 回到程序打开 PDF；此后 Worker 断点才会命中。

如果需要观察 PDFium 的 C# 适配层，再在 `PdfNativeAdapter.RenderPage()`、`GetPageText()` 附近下断点。不要尝试单独直接启动 Worker，它需要主程序传入管道名称并完成握手。

### 推荐断点和观察项

- `Program.cs` 的 `switch (request.MessageType)`：观察收到的消息类型。
- `WorkerDocumentService.Open()`：观察新 `DocumentId` 和文档字典。
- `PdfiumNativeAdapter.Open()`：观察 PDFium 句柄是否成功创建。
- `PdfiumNativeAdapter.RenderPage()`：观察页面宽高、缩放、旋转和像素尺寸。
- `WorkerDocumentService.Render()`：观察共享内存名称与长度。
- `WorkerDocumentService.ReleaseSharedMemory()`：确认主程序使用后释放映射。
- `PdfiumNativeAdapter.ExtractPageText()`：观察字符数量和每个字符的边界。
- `Dispose()`：观察文档、页面、文字页和共享内存由谁释放。

### 必须理解的资源所有权

- PDFium 文档、页面和文字页句柄由 Worker 创建并释放。
- Worker 创建内存映射文件，主程序读取完成后发送释放请求。
- 主程序关闭文档时，Worker 释放该文档的原生资源。
- 即使渲染结果已经过期，主程序仍必须通知 Worker 释放共享内存。

### 完成标准

能够说明为什么主程序崩溃风险较低：PDFium 崩溃首先影响 Worker，而不是 WPF 进程。

## 10. 第五轮：主程序侧 IPC 客户端

### 阅读顺序

1. `Application/PdfWorker/IPdfWorkerClient.cs`
2. `Infrastructure/PdfWorker/PendingRequestRegistry.cs`
3. `Infrastructure/PdfWorker/PdfWorkerClient.cs`

### 推荐断点

- `PdfWorkerClient.StartAsync()`：观察进程路径、管道名和握手。
- `SendRequestAsync()`：观察生成的 RequestId。
- `PendingRequestRegistry` 的注册与完成方法：观察等待中的请求。
- `ReceiveLoopAsync()`：观察响应如何按 RequestId 交还调用者。
- `OpenDocumentAsync()`、`RenderPageAsync()`、`GetPageTextAsync()`：观察业务方法如何包装协议消息。
- `ReportDisconnected()`：观察进程退出和管道断开的分类。

### 双进程跟踪方法

在 `PdfWorkerClient.SendRequestAsync()` 和 Worker `Program.cs` 的 `switch` 同时下断点：

1. 主程序断在发送前，记下 RequestId 和 MessageType。
2. 继续运行，Worker 断在接收处，确认 RequestId 相同。
3. Worker 写回响应后继续。
4. 主程序响应循环命中，确认通过 RequestId 找到原来的等待任务。

### 完成标准

能够解释为什么响应可以乱序到达，以及 `PendingRequestRegistry` 如何防止返回给错误的调用者。

## 11. 第六轮：阅读主链 ReaderViewModel

`ReaderViewModel.cs` 是 V0.1 最复杂的单个文件，应按调用链分段阅读，不能从第一行读到最后一行。

### 路径 A：打开文档

断点顺序：

1. `MainWindow.xaml.cs` 的打开文件事件。
2. `ReaderViewModel.OpenDocumentAsync()`。
3. `PdfWorkerClient.OpenDocumentAsync()`。
4. Worker 的 `OpenDocumentRequest` 分支。
5. 返回 `ReaderViewModel.RenderCurrentPageAsync()`。

观察：`filePath`、`DocumentId`、`PageCount`、`CurrentPageIndex`、`IsOpening`。

### 路径 B：渲染页面

断点顺序：

1. `ReaderViewModel.RenderCurrentPageAsync()`。
2. `PageRenderCache.TryGet()`。
3. `PdfWorkerClient.RenderPageAsync()`。
4. Worker 的 `RenderPageRequest` 分支。
5. `ReaderViewModel` 读取共享内存并创建 WPF 位图的位置。
6. `ReleaseSharedMemoryAsync()`。

观察：渲染版本号、RequestId、页码、缩放、旋转、缓存键、共享内存名称、结果是否过期。

快速连续点击“下一页”可以观察旧请求完成后被丢弃，但共享内存仍被释放。

### 路径 C：文字选择

断点顺序：

1. `MainWindow.xaml.cs` 的鼠标按下、移动和松开事件。
2. `ReaderViewModel.BeginSelectionAsync()`。
3. `ReaderViewModel.UpdateSelection()`。
4. `ReaderViewModel.HitTestSelection()`。
5. `TextSelectionService.CreateSelection()`。
6. `ReaderViewModel.CompleteSelection()`。

观察：鼠标起点、拖动距离阈值、视图坐标、PDF 坐标、字符索引、规范化原文和选区矩形。

分别做两次练习：只单击文字时不应创建选区；按住并拖动超过阈值时才应创建选区。

### 完成标准

能够独立画出“打开 PDF”和“选择文字”两条调用链，并指出异步边界和资源释放位置。

## 12. 第七轮：WPF、MVVM 和数据绑定

### 阅读顺序

1. `App.xaml`
2. `App.xaml.cs` 的 `OnStartup()` 和服务注册部分
3. `MainWindow.xaml`
4. `MainWindow.xaml.cs`
5. `Commands/AsyncCommand.cs`
6. `ReaderViewModel.cs` 的属性、命令和 `OnPropertyChanged()`

### 重点理解

- View 是 XAML 界面，不负责 PDF 业务。
- ViewModel 提供属性和命令，通过 `DataContext` 连接界面。
- `INotifyPropertyChanged` 通知 WPF 某个属性已经变化。
- `AsyncCommand` 把异步方法包装成按钮可绑定的命令。
- DI（Dependency Injection，依赖注入）在 `App.xaml.cs` 中把接口和实现组合起来。

### 推荐断点

- `App.OnStartup()`：观察服务构建、Worker 启动和窗口显示顺序。
- `ConfigureServices()` 附近：观察 `IPdfWorkerClient`、`ITranslationService` 的实际实现类型。
- `AsyncCommand.Execute()`：观察按钮点击如何进入异步业务方法。
- `ReaderViewModel.OnPropertyChanged()`：使用条件断点 `propertyName == "CurrentPageNumber"`。

### 调试练习

点击“下一页”，在调用堆栈中依次找到 XAML 命令、`AsyncCommand`、ViewModel 和渲染方法。不要进入 WPF 框架内部代码。

### 完成标准

能够解释为什么按钮可用状态不应该直接在 `MainWindow.xaml.cs` 中手工维护。

### 推荐断点

### 安全提醒

- 不要在断点条件、监视窗口、截图或聊天中粘贴真实 API 密钥。

## 16. 测试代码的阅读方法

不要把测试只当作“通过或失败”的按钮。测试是最短的使用示例：

- Arrange：准备输入和伪造依赖。
- Act：调用一个方法。
- Assert：说明正确结果是什么。

建议顺序：

1. Domain 测试：熟悉记录类型。
2. Application 测试：熟悉状态、坐标、选区、缓存和翻译规则。
3. Protocol 测试：熟悉序列化和消息边界。
4. Integration 测试：熟悉 Worker、PDFium、ViewModel 和翻译提供器的组合。

遇到不懂的正式方法时，先对方法名执行“查找所有引用”，优先阅读调用它的测试，再回到实现。

## 17. 一条完整调用链

打开并显示一页 PDF 的核心路径：

```text
MainWindow 打开按钮
→ ReaderViewModel.OpenDocumentAsync
→ IPdfWorkerClient.OpenDocumentAsync
→ PdfWorkerClient 封装 OpenDocumentRequest
→ 命名管道
→ Worker Program 分发请求
→ WorkerDocumentService.Open
→ PdfiumNativeAdapter 打开 PDF
→ OpenDocumentResponse 返回
→ ReaderViewModel.RenderCurrentPageAsync
→ PdfWorkerClient 发送 RenderPageRequest
→ WorkerDocumentService 创建共享内存
→ ReaderViewModel 读取 BGRA 位图
→ WPF Image 显示页面
→ ReleaseSharedMemoryRequest
```

选中文字并翻译的核心路径：

```text
MainWindow 鼠标事件
→ ReaderViewModel 获取页面字符
→ CoordinateTransformer 转换坐标
→ TextSelectionService 命中并创建选区
→ TranslationPanelViewModel 更新可编辑原文
→ 用户点击开始翻译
→ TranslationService 检查设置、隐私和凭据
→ ITranslationProvider
→ DeepSeekTranslationProvider 读取 SSE 分片
→ TranslationPanelViewModel 批量刷新译文
```

## 18. 调试问题的定位顺序

页面没有显示：

1. `ReaderViewModel` 是否创建了正确渲染请求；
2. `PdfWorkerClient` 是否发送成功；
3. Worker 是否收到请求；
4. PDFium 是否成功生成位图；
5. 共享内存是否存在且长度正确；
6. WPF 位图是否在界面线程更新；
7. 结果是否因版本过期被主动丢弃。

文字选取错误：

1. 页面文字层是否存在；
2. 字符边界是否正确；
3. DPI、缩放和旋转参数是否进入 `PageTransformContext`；
4. View→PDF 坐标是否正确；
5. 拖动阈值是否达到；
6. 起止字符索引是否正确；
7. 规范化是否错误删除了字符。

翻译失败：

1. 是否由用户主动发起；
2. 网络隐私开关是否允许；
3. 设置是否加载；
4. 凭据名称是否正确且密钥存在；
5. API 地址是否规范化；
6. HTTP 状态属于认证、限流还是服务器错误；
7. SSE 内容是否被解析；
8. 请求是否被用户、新请求或超时取消。

Worker 恢复失败：

1. 是进程退出还是管道断开；
2. 是否只触发一次重启；
3. 旧连接和旧等待请求是否清理；
4. 新 Worker 是否握手和 Ping 成功；
5. 原 PDF 路径是否仍可访问；
6. 新 `DocumentId` 是否替换旧值；
7. 阅读视图快照是否重新应用。
