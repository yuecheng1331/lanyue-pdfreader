## 总体结论

V0.5 已经从“技术验证型 PDF 阅读器”发展为“可持续日常使用的本地阅读器”。专门的 V0.5 设计中列出的 18 项完成标准均已达成，并额外提前实现了原计划属于 V1.0 的多文档标签页基础。

但它还不是最初总体规划中的“正式个人阅读版本”：连续阅读、目录缩略图、打印、密码 PDF、高级批注、页面/全文翻译、朗读和安全报告仍需在 V1.0 或后续 1.x 版本中完成。

## 二、与专门 V0.5 设计的对照

专门的 V0.5 设计主动缩小了早期总体规划的范围，核心目标是“最近文件＋状态恢复＋搜索＋本地批注＋可靠发布”。这个版本冻结是合理的，也已经完整实现。  
:codex-file-citation{path="C:/Users/29295/Desktop/pdfreader/docs/LocalPdfReader V0.5设计文档.docx" artifact_kind="document"}

主要设计调整包括：

- 启动行为从“自动打开上次 PDF”调整为始终进入主页，避免失效文件和旧会话阻塞用户。
- 阅读状态仍然保存，但由用户从最近文件或文件选择重新打开后恢复。
- 多标签页因实际业务需求提前从 V1.0 插入 V0.5。
- 批注采用本地 SQLite，不修改原 PDF，优先保证安全与可靠性。
- 搜索不建立永久全文索引，避免隐私、数据库体积和文件变更问题。
- 打开 PDF 采用失败回滚：打开失败不产生标签页或最近记录，也不会因关闭无效标签导致崩溃。

这些并不是对设计的简单删减，而是人工测试驱动出的更合理产品行为。

## 四、当前版本最重要的成功

### 1. 架构路线得到验证

最初的“WPF 主程序＋独立 PDF Worker＋命名管道＋共享内存”架构不仅能打开和渲染 PDF，也成功承载了多标签、搜索、取消、崩溃恢复和批注恢复。

这意味着 V1.0 无需推倒重来。

### 2. 本地数据模型形成稳定基础

文档指纹、SQLite 模式、仓储接口、阅读状态和批注模型已经建立。V1.0 的收藏、会话恢复、批注筛选和阅读笔记都可以在此基础上扩展。

### 3. 异常路径不再是附属功能

V0.5 解决了几类关键问题：

- PDF 打开失败不产生幽灵标签。
- 关闭无效标签不会导致程序崩溃。
- Worker 崩溃后可以重建文档会话。
- 数据库不可用不阻止基本阅读。
- 多标签搜索不会互相污染。
- 旧渲染结果不会覆盖新状态。

### 4. 项目已经具备版本发布能力

V0.5 建立了自动测试、依赖检查、敏感文件检查、自包含发布、解压回验和隔离启动验证。这使后续版本不再需要临时手工拼装发布包。

### 阶段 A：V1.0 基线与架构整理

- 冻结 `v0.5.0`。

### 阶段 B：标签页与会话管理

- 收藏文件。
- 恢复误关闭标签。
- 保存上次标签会话。
- 启动时默认仍显示主页，由用户选择“恢复会话”。
- 定义重复打开同一 PDF 的行为。
- 完善大量标签下的资源释放。

### 阶段 C：正式阅读模式

- 连续滚动。
- 单页、双页和双页连续。
- 封面单独显示。
- 全屏和快捷键。
- 平滑滚动与触控板缩放。
- 页面历史前进/后退。

这是 V1.0 最应该优先提升的“阅读器本体”。

### 阶段 D：目录、缩略图和书签

- Worker 读取 PDF 目录。
- 缩略图按可见区域后台生成。
- 点击目录/缩略图跳转。
- 缩略图缓存与内存限制。
- 与标签页、旋转和阅读模式隔离。

### 阶段 E：文件兼容与输出

- 密码 PDF。
- 外部文件修改检测。
- 打印。
- 保存和另存为。
- 打开外部链接前确认。
- 明确是否允许修改原 PDF。

建议先完成“另存为”和打印，再决定是否直接写回原文件。

### 阶段 F：批注 V2

- 下划线、删除线和波浪线。
- 页面便签。
- 撤销与重做。
- 按颜色、类型、页码筛选。
- 导出 Markdown/TXT。
- 最后单独评估标准 PDF 批注写入。

“本地批注扩展”和“写入 PDF”应分成两个阶段，因为后者涉及签名失效、备份和 Edge/Adobe 兼容测试。

### 阶段 G：翻译 V2

- 页面和指定范围翻译。
- 长文本分段。
- 字符量与费用提示。
- 翻译历史。
- 翻译缓存。
- 个人术语表。
- 上下文与专业预设增强。

全文翻译必须在隐私提示、取消、断点恢复和费用控制完成后再开放。

### 阶段 H：朗读、安全、性能与发布

- Windows TTS 朗读。
- 基础无障碍。
- 密码和异常 PDF 分类。
- Worker 内存、页面尺寸和任务时间限制。
- 基础安全报告。
- 数百页、数百 MB PDF 性能验证。
- V1.0 发布门禁和便携包验收。

## 七、建议的 V1.0 范围控制

不建议把最初总体规划中所有 V1.0 功能一次塞进一个开发周期。更合理的优先级是：

- P0：架构拆分、会话、正式阅读模式、目录缩略图、密码 PDF、打印、性能与恢复。
- P1：批注筛选导出、基础 PDF 写入、翻译历史和长文本翻译。
- P2：手写图形、全文翻译、术语表、TTS。
- 继续延期：OCR、论文结构识别、全文问答、数字签名验证、高级安全扫描、插件和 GPU 加速。

## 八、V1.0 立项前需要冻结的五个决定

1. 启动时是主页提供“恢复会话”，还是自动恢复全部标签。
2. 批注继续只存本地，还是 V1.0 必须写入标准 PDF。
3. 连续阅读、双页阅读和目录缩略图的优先顺序。
4. 页面/全文翻译是否属于 V1.0 必须功能，能接受怎样的费用和隐私提示。
5. 打印、密码 PDF、朗读三者的业务优先级。

总体来看，V0.5 已经成功验证了产品核心和架构基础。V1.0 的重点不应是继续堆积零散按钮，而是先稳定“文档会话模型”，再依次扩展阅读、导航、输出、批注和翻译能力。
