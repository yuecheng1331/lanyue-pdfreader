## 本次决定

V1.0 阶段 C「正式阅读模式」的当前未提交实现已经回退。原因是连续滚动、双页布局、缩放、当前页同步、文本选择之间的状态关系没有形成可靠闭环，继续在现有实现上修补的风险高于重新设计。

回退仅作用于阶段 C 及其后续的未提交代码和测试改动；阶段 A、B 已提交的功能保留不变。`Backup/`、`UpgradeLog.htm` 与原有 `docs/` 内容未删除。

## 当前可用范围

阶段 B 基线包含以下已提交能力：

- V0.5 已发布能力：本地 PDF 打开、渲染、缩放、旋转、文本选择与翻译、最近文件、SQLite 持久化、搜索、本地批注等。
- V1.0 阶段 A：对阅读器职责进行初步拆分，加入渲染协调与阅读状态协调组件。
- V1.0 阶段 B：多文档标签页和会话持久化基础；包含标签切换、关闭、拖动排序、恢复关闭标签的基础数据模型与本地存储。

本基线**不包含**本轮尝试的单页连续、双页、双页连续、固定 Canvas 缩放、阅读模式切换控件及相关测试。后续重新实现时，不应从任何已回退的代码片段复制逻辑。

## 工作区状态约定

- 已跟踪代码应与提交 `f04e47e` 一致。
- 工作区仍可能显示 `Backup/`、`UpgradeLog.htm`、`docs/` 为未跟踪文件；它们不是本次回退对象，也不应在无明确要求时加入提交。
- 本轮新增但未跟踪的 `src/LocalPdfReader.App/ReadingModeViewModels.cs` 已移除，避免残留实现进入后续构建。
- 后续每个阶段在用户完成阶段测试并明确要求进入下一阶段后，再创建本地 Git 提交；不要在阶段内自动提交。

## 重新开始阶段 C 前的门槛

1. 先在阶段 B 基线上完成 Release 构建与现有相关测试，确认回退后的基线可用。
2. 将「阅读模式」拆成独立的小阶段，先只确定数据模型和布局状态，再做单页连续，最后才做双页。
3. 每一阶段只处理一种滚动和布局关系，并在阶段末由用户集中测试；不要在阶段内要求用户反复验证。
4. 缩放、当前页、可见页预加载与文本选择必须共享同一个明确坐标系，不能各自按页码或位图尺寸推导。
