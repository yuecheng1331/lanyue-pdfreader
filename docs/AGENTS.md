AGENTS.md

1.项目概述

本项目是一款面向Windows 11的本地PDF阅读器。

当前只开发V0.1技术验证版本，目标闭环为：

打开本地PDF→独立工作进程加载PDF→渲染页面→翻页、缩放和旋转→选择并复制文字→调用DeepSeek API完成翻译→显示和复制译文。

当前确定的技术路线：

•	开发语言：C#；

•	运行平台：.NET；

•	界面框架：WPF（Windows Presentation Foundation，Windows桌面界面框架）；

•	界面架构：MVVM（Model-View-ViewModel，模型-视图-视图模型）；

•	PDF引擎：PDFium；

•	进程结构：主程序LocalPdfReader.App.exe＋PDF工作进程LocalPdfReader.PdfWorker.exe；

•	控制通信：Windows命名管道；

•	页面位图传输：内存映射文件；

•	配置存储：JSON；

•	API密钥存储：Windows凭据存储；

•	翻译接口：DeepSeek兼容API；

•	目标平台：Windows 11 x64。

V0.1不开发：

•	PDF批注；

•	全文搜索；

•	目录和缩略图侧栏；

•	SQLite数据库；

•	完整多标签页；

•	全文翻译；

•	OCR；

•	数字签名；

•	高级安全扫描；

•	插件系统；

•	GPU加速；

•	PDF表单；

•	完整PDF编辑。

不得擅自扩展V0.1范围。

3.基本协作原则

3.1采用增量开发

每次只完成一个边界明确、可以验证的小任务。

一个任务通常应满足：

•	有明确目标；

•	涉及有限数量的文件；

•	能够独立编译或测试；

•	完成后可以通过具体操作验证；

•	适合形成一次独立Git提交。

不得一次性生成整个项目或整个开发阶段的全部代码。

3.2先建立最小闭环

优先实现能够运行的最小路径，再增加抽象、错误处理和优化。

推荐顺序：

1.建立解决方案； 2.启动主程序和工作进程； 3.完成命名管道握手； 4.打开PDF； 5.渲染第一页； 6.传输并显示位图； 7.实现基础翻页和缩放； 8.提取文字； 9.实现文字选择； 10.接入DeepSeek翻译； 11.补充恢复、日志和测试。

不得在核心闭环尚未完成时提前实现低优先级功能。

3.3避免过度设计

只为已经确定的V0.1需求创建接口和模块。

可以为后续替换PDFium封装或翻译提供者保留合理接口，但不得创建没有当前调用方的复杂框架。

发现以下情况时，应主动指出可能存在过度设计：

•	接口只有一个简单实现且没有明确替换需求；

•	为未来功能提前创建大量空类；

•	使用复杂设计模式解决简单问题；

•	为尚未确定的V1.0功能修改V0.1核心结构；

•	单个简单功能被拆分为过多项目或服务。

3.4保持可运行状态

完成每个任务后，解决方案应尽量保持：

•	可以还原依赖；

•	可以编译；

•	可以运行已有功能；

•	已有测试可以通过；

•	不包含明显未处理的编译错误；

•	不依赖用户手动补全关键代码。

不能为了展示结构而提交大量不可运行的占位实现。

实施时遵守：

•	优先完成最小可运行实现；

•	一次只修改当前任务相关内容；

•	不顺便重构无关模块；

•	不擅自升级大范围依赖；

第四步：运行验证

修改后应尽可能运行：

•	适合自动执行的最小运行验证。

第五步：解释结果

完成后必须说明：

•	实际修改了什么；

•	关键代码如何工作；

•	用户本次应该掌握什么；

5.教学要求

5.1解释当前需要的概念

只解释完成当前任务所必需的概念。

例如：

•	建立WPF项目时解释应用入口、窗口、数据绑定和DataContext；

•	使用MVVM时解释View、ViewModel和命令的职责；

•	使用依赖注入时解释注册、解析和生命周期；

•	使用命名管道时解释客户端、服务端、消息边界和请求匹配；

•	使用内存映射文件时解释它为什么适合传输页面位图；

•	使用PDFium时解释原生资源句柄和释放问题。

不得在当前任务中提前展开大量后续知识。

5.2解释设计理由

对于重要结构，需要说明“为什么这样做”，而不只是说明“代码做了什么”。

重点解释：

•	为什么主程序不直接引用PDFium；

•	为什么PDF解析放在独立工作进程；

•	为什么领域层不引用WPF类型；

•	为什么请求需要RequestId；

•	为什么异步任务需要取消；

•	为什么API密钥不能保存在JSON中；

•	为什么页面位图不适合通过JSON传输；

•	为什么旧渲染结果需要丢弃。

5.3解释缩写和术语

首次使用英文缩写时必须给出完整英文和中文含义。

例如：

•	MVVM（Model-View-ViewModel，模型-视图-视图模型）；

•	IPC（Inter-Process Communication，进程间通信）；

•	DPI（Dots Per Inch，每英寸点数）；

•	DI（Dependency Injection，依赖注入）；

•	DTO（Data Transfer Object，数据传输对象）。

不能假设用户已经理解专业缩写。

5.4提供可操作的调试方法

出现问题时，不得只给出修复后的代码，还需要说明：

•	如何复现问题；

•	应在哪里设置断点；

•	需要观察哪些变量；

•	应查看哪一段日志；

•	如何区分界面问题、通信问题和PDFium问题；

•	如何验证修复确实生效。

5.5避免黑箱式生成

对于关键代码，不得只生成代码而不说明入口和调用关系。

至少应说明：

•	谁调用该代码；

•	输入是什么；

•	输出是什么；

•	错误如何返回；

•	资源由谁释放；

•	取消操作如何传递；

•	相关测试覆盖什么。

不需要逐行解释机械性代码。

6.代码实现要求

6.1语言和版本

•	使用C#；

•	使用当前项目已经确定的.NET版本；

•	启用可空引用类型；

•	启用隐式using时保持全项目一致；

•	不得未经说明更改目标框架；

•	不得无理由使用预览语言特性。

6.2异步规范

涉及以下操作时优先使用异步方法：

•	文件访问；

•	命名管道通信；

•	HTTP请求；

•	页面渲染调度；

•	工作进程启动和停止；

•	配置读写。

异步方法名称使用Async后缀。

可取消的长时间操作必须接受CancellationToken。

不得使用：

•	.Result；

•	.Wait()；

•	无必要的async void；

•	在WPF界面线程执行阻塞式网络或文件操作。

事件处理器是允许使用async void的例外，但业务逻辑应下沉到可等待的方法中。

6.3资源管理

以下资源必须明确释放：

•	PDF文档句柄；

•	PDF页面句柄；

•	PDF文本页面句柄；

•	命名管道；

•	内存映射文件；

•	文件流； -取消令牌源；

•	工作进程；

•	HTTP响应流。

优先使用：

•	using；

•	await using；

•	IDisposable；

•	IAsyncDisposable；

•	try/finally。

每次引入原生资源时，需要说明资源所有权。

6.4错误处理

禁止空的catch。

错误处理应区分：

•	用户输入错误；

•	文件访问错误；

•	PDF格式错误；

•	密码错误；

•	工作进程错误；

•	网络错误；

•	API认证错误；

•	请求超时；

•	用户取消。

用户界面显示可理解的信息，技术细节写入日志。

不得向用户直接显示完整堆栈。

6.5日志

日志不得记录：

•	DeepSeek API密钥；

•	HTTP授权头；

•	PDF密码；

•	完整待翻译原文；

•	完整译文；

•	用户未允许记录的敏感内容。

关键异步任务日志应包含：

•	RequestId；

•	DocumentId；

•	模块名称；

•	任务状态；

•	错误码；

•	耗时。

6.6命名

命名应表达职责，不使用无意义缩写。

推荐：

PdfWorkerClient

PageRenderService

TranslationRequest

DocumentSession

CoordinateTransformer

避免：

Mgr

Util

Helper

Data1

TempService

DoWork

HandleThing

Helper仅用于真正无状态、无领域含义的通用小功能。

6.7注释

注释解释设计意图、边界条件和特殊原因，不重复代码表面含义。

需要注释的内容：

•	坐标系方向；

•	页面旋转变换；

•	原生资源所有权；

•	命名管道消息边界；

•	旧渲染结果丢弃条件；

•	安全限制；

•	非直观的PDFium行为。

简单属性和明显代码不需要逐行注释。
