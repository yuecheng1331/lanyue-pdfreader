## 2. 已完成的产品闭环

V0.1 已形成以下完整路径：

打开本地 PDF → 独立 Worker 加载和渲染 → 翻页、缩放、适配与旋转 → 拖动选择文字 → 编辑原文 → 调用 DeepSeek 流式翻译 → 编辑或复制译文。

阅读能力：

- 打开普通本地 PDF，支持中文及空格路径和只读文件。
- 上一页、下一页、页码跳转、放大、缩小、适合页面、适合宽度和四方向旋转。
- 页面位图缓存和最新渲染请求优先，避免快速操作时旧结果覆盖新结果。
- PDFium 文字层提取、拖动选区、原文复制和常见英文跨行断词合并。
- 单击文字或空白不会误建选区，也不会无故清空翻译上下文。

翻译能力：

- DeepSeek OpenAI 兼容接口、连接测试、流式输出、停止、超时和重新翻译。
- 直译、流畅、学术论文、计算机科学、医学和自定义风格。
- 原文与译文均可编辑，用户可删除页眉、页码、资助信息等误选内容。
- 同一时间仅保留一个翻译请求；错误、限流或超时不会自动重试，避免意外费用。
- 翻译提示词将 PDF 内容视为普通待翻译文本，降低提示注入风险。

安全与恢复：

- API 密钥保存在 Windows 凭据存储，不进入 JSON、代码、日志、测试或发布包。
- 设置按 Windows 用户存放在本地应用数据目录，不随程序发布。
- PDFium 仅存在于独立 Worker，主程序不直接加载原生 PDF 引擎。
- Worker 异常退出后，主程序检测断开、自动重启、重新握手并恢复文档、页码、缩放、旋转和适配模式。
- 统一用户错误提示与稳定错误代码；技术详情进入本地诊断日志，不主动记录密钥、完整原文或完整译文。

发布与质量：

- 固定 PDF 测试集覆盖纵向、横向、多页、断词、无文字层、只读、损坏文件和四方向旋转。
- Windows 11 x64 自包含发布同时打包主程序、Worker、.NET 运行时和 PDFium。
- 发布脚本检查必要文件并阻止密钥、用户设置、证书和日志混入 ZIP。
- 最终 ZIP 已完成解压启动验证；关闭系统级 .NET 查找后，主程序和 Worker 仍可启动。

## 3. 当前架构资产

- `LocalPdfReader.App`：WPF 界面、ViewModel、命令和用户交互。
- `LocalPdfReader.Application`：阅读、选择、翻译、凭据与设置的应用层接口和业务规则。
- `LocalPdfReader.Domain`：不依赖 WPF、HTTP 或 PDFium 的领域模型。
- `LocalPdfReader.Infrastructure`：DeepSeek、Windows 凭据、设置和日志等外部实现。
- `LocalPdfReader.PdfProtocol`：主程序与 Worker 的命名管道消息契约。
- `LocalPdfReader.PdfWorker`：PDFium 文档、页面、文字层和位图资源管理。
- `LocalPdfReader.Shared`：受控的跨项目基础类型。
- 四个测试项目：Domain、Application、PdfProtocol 和 Integration。

现有分层已为 V0.5 保留两个重要扩展点：翻译提供器接口允许增加新引擎；PDF Worker 边界允许继续增强 PDF 能力而不把原生库引入界面进程。

## 4. V0.1 明确边界

- 不支持密码保护 PDF、OCR、全文搜索、目录、缩略图侧栏、批注和完整 PDF 编辑。
- 不支持多标签页、全文翻译、PDF 表单、数字签名、插件系统和 GPU 加速。
- 当前只有 DeepSeek 正式翻译实现，尚无网易等低资费提供器。
- 双栏、复杂公式、透明图层、特殊嵌入字体和高 DPI 主要依赖人工验证。
- 发布物是未签名的便携 ZIP，不是安装程序，Windows 可能显示未知发布者。
- 尚未在一台完全没有开发环境的独立 Windows 11 x64 设备上人工验收；当前以禁用系统级 .NET 查找的本机测试替代。

## 5. V0.5 建议规划顺序

### V0.5-A：冻结基线与需求选择

- 为 `29858ab` 建立 `v0.1.0` 标签，并在界面或“关于”页显示正式版本号。
- 从候选功能中确定 V0.5 必做、可选和不做范围，避免同时扩展阅读、翻译和安装三个方向。
- 确定设置升级策略，保证 V0.1 用户升级后凭据和偏好不丢失。

### V0.5-B：文档会话与导航

- 最近打开文件、启动后恢复上次文档和阅读位置。
- 在目录、缩略图和全文搜索中选择一项先完成最小闭环；建议优先全文搜索或目录，暂不同时开发三项。
- 为大型文档明确缓存上限和后台加载策略。

### V0.5-C：多翻译引擎与费用控制

- 基于现有 `ITranslationProvider` 增加网易等低资费引擎，每个提供器使用独立凭据名和设置。
- 在发起请求前显示提供器、字符数或估算用量；保留“只由用户点击才联网”和“不自动重试”的原则。
- 为不同引擎分别实现认证、错误分类、超时、流式能力和隐私说明，不能只替换 URL。

### V0.5-D：复杂论文文字选择

- 增强双栏阅读顺序、页眉页脚识别和段落级选择。
- 建立真实论文回归集，但不得提交有版权或含敏感信息的论文原件。
- OCR 若进入 V0.5，应作为独立技术验证阶段，不与普通文字层选择混在一次任务中。

### V0.5-E：分发与升级

- 决定继续便携 ZIP，还是增加 MSIX/安装程序；代码签名应单独评估证书费用和发布流程。
- 在干净的 Windows 11 x64 虚拟机或测试设备完成安装、启动、升级、卸载和 SmartScreen 验证。
- 增加发布版本清单、校验值和可重复构建记录。

### V0.5-F：总体验收

- 先执行完整自动测试，再按阅读、选择、翻译、恢复、设置迁移和发布逐项人工验收。
- 真实翻译 API 只做少量限额测试；其余使用伪造响应，避免重复费用。
- 每个阶段人工通过后形成独立提交，最后建立正式版本标签。

## 6. 开始 V0.5 前需要决定的问题

1. V0.5 的首要价值是“阅读导航”“复杂论文选择”“更多翻译引擎”还是“正式安装发布”？
2. 是否把最近文件和阅读位置持久化列为必做？
3. 网易等翻译接口的具体产品、计费方式和是否支持流式输出是什么？
4. OCR 是否进入 V0.5；如果进入，是否接受额外模型体积和性能开销？
5. 是否需要安装程序和代码签名，还是继续使用便携 ZIP？
6. V0.5 是否需要兼容 V0.1 设置与凭据，还是允许清空后重新配置？

建议先回答第 1 项并冻结 V0.5 范围，再把后续内容拆分为可独立验收和提交的小阶段。
